Project Goal

새로운 확장자 .ctrlc를 실행 가능한 언어로 만듭니다

입력

  • .ctrlc source file
  • prompt에서 입력한 한 줄 코드
  • debug 대상으로 실행할 프로그램
→

결과

  • source를 Token과 Expr/Stmt AST로 구조화
  • Checker로 scope, class, import 규칙을 실행 전에 확인
  • Executor가 Environment를 갱신하며 출력과 오류를 생성

Architecture

Shell은 실행 방식을 나누고, Pipeline은 같은 순서로 해석합니다

Shell Modes

Common Interpreter Pipeline

01Tokenizer문자열을 의미 있는 token stream으로 분리

Design Patterns

구현에 적용한 디자인 패턴

반복되는 구현 문제를 어떤 구조로 정리했는지 패턴별로 연결해서 설명합니다.

Design Patterns

디자인 패턴은 복잡도가 커지는 지점에 적용했습니다

AST 구조 Composite-style AST Stmt와 Expr node가 서로를 포함하는 tree가 되어 Checker와 Executor가 같은 구조를 따라갑니다. expr.py · statement.py
호출 규칙 Callable Adapter native 함수, 사용자 함수, class 생성자를 모두 arity()와 call() 규칙으로 맞춥니다. runtime.py · executor.py
생성 책임 Factory-like Creation source 입력은 Tokenizer와 Assembler를 지나 token과 AST node로 만들어집니다. tokenizer.py · assembler.py
Shell 명령 Command Pattern prompt 입력을 command 객체로 바꾸고, Shell은 execute(shell)만 호출합니다. commands.py · shell.py

Pattern · Composite

Composite-style AST: 코드를 트리 구조로 다룹니다

적용 지점

조건문, 반복문, 함수 본문처럼 내부에 다른 문장과 식이 들어가는 구조를 Stmt/Expr node tree로 표현했습니다.

문법 추가 위치가 명확함
Checker와 Executor가 같은 AST를 사용함
중첩 구조를 자연스럽게 표현함

BP Case 3 · Refactoring

Shell Command Refactoring

Before

  • PromptShell 안에서 명령 분기가 커짐
  • 기능 추가 시 기존 흐름을 계속 건드림
  • 명령별 테스트 범위가 넓어짐

After

  • 명령을 Command 객체로 분리
  • PromptShell은 command 선택과 실행만 담당
  • 새 shell 기능을 독립 단위로 추가
